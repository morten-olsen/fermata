## Design rationale

Bottom sheets are one of only two modal patterns in Fermata (the other is the full-screen Now Playing overlay). They're used sparingly — only for track actions and the playback queue.

## Why not a library?

Fermata uses a custom bottom sheet instead of `@gorhom/bottom-sheet` or similar. The reasons:

<div style={{ padding: 20, backgroundColor: '#141416', borderRadius: 12, margin: '24px 0' }}>
  <ol style={{ paddingLeft: 20, lineHeight: 2, color: '#9898A3', fontSize: 14, margin: 0 }}>
    <li><strong style={{ color: '#E8E8ED' }}>Minimal needs.</strong> Fermata only uses sheets for two things — track actions and the queue. No snap points, no half-expand, no nested scrolling.</li>
    <li><strong style={{ color: '#E8E8ED' }}>Controlled animation.</strong> The spring tuning matches the rest of the app's motion language exactly.</li>
    <li><strong style={{ color: '#E8E8ED' }}>Fewer dependencies.</strong> ~120 lines of code vs a library with gesture handlers, worklets, and platform-specific edge cases.</li>
  </ol>

## Variants

### With scrollable content
<Canvas of={BottomSheetStories.WithScrollableContent} />

The sheet's content area grows with its children. For longer lists (like queue), the parent wraps content in a `ScrollView` or `FlatList`.

## Usage

```tsx
<BottomSheet visible={showSheet} onDismiss={() => setShowSheet(false)}>
  <View style={{ paddingHorizontal: 20 }}>
    {/* Sheet content */}
  </View>
</BottomSheet>
```

The sheet handles its own animation lifecycle — just toggle `visible`. Safe area padding is applied automatically at the bottom.
